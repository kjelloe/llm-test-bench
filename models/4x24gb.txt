# 4× 24 GB VRAM model set — RTX 4090 + 3× RTX 3090 (96 GB total, PCIe)
# All 4-GPU models use tensor_split=1|1|1|1 (equal layer distribution across all four cards).
#
# Format: <ollama-name>  [<gguf-file>  [<params>]]  [hf:<owner/repo>]
#   gguf-file  GGUF filename relative to $LLAMA_MODELS_DIR (for --backend llama-server)
#   params     comma-separated llama-server flags
#   hf:        HuggingFace repo for fetch-hf.sh download helper
#
# Hardware: RTX 4090 (~1008 GB/s) + 3× RTX 3090 (~936 GB/s each); all connected by PCIe.
# No NVLink between any pair. For dense tensor-parallel, bandwidth scales as sum across cards
# for generation (memory-bound), but inter-GPU activation transfers add PCIe latency per hop.
# Use: ./gpu-mode.sh multi  before running (sets CUDA_VISIBLE_DEVICES to all 4 GPUs).
#
# What 96 GB unlocks vs 72 GB (3x24gb.txt):
# - Dense 32B: 19.4 GB/GPU free → max_ctx=262144 trivial; full 33-task suite feasible
# - Dense 70B: 14 GB/GPU free → max_ctx=131072 feasible (was 65536 at 72 GB)
# - qwen2.5:72b-q4: 13.75 GB/GPU → max_ctx=131072 (was 65536 at 72 GB)
# - gpt-oss:120b: 9 GB/GPU free → max_ctx=65536 (was 32768 at 72 GB)
# - qwen3-next:80b: 13.75 GB/GPU free → VRAM constraint for context_256k CLEARS;
#   architecture limit unknown → set max_ctx=262144 and check: CTX_TRUNCATED = arch limit,
#   PASS/TESTS_STILL_FAIL = architecture supports 262k (then fix max_ctx=131072 in 2x24gb.txt)
# - llama4-scout:17b: 9 GB/GPU free → max_ctx=65536 (was 32768 at 72 GB)
# - Small/MoE models (<25 GB): no meaningful speed gain from 4th GPU vs 3rd; run single-GPU
# - Dense 70B Q8_0 is SLOWER than Q4_K_M on 4-way PCIe: memory-bandwidth bound means
#   doubling weight size roughly halves tok/s; quality gain does not justify the penalty
