# 3× 24 GB VRAM model set — RTX 4090 + 2× RTX 3090 (72 GB total, PCIe)
# All 3-GPU models use tensor_split=1|1|1 (equal layer distribution across all three cards).
#
# Format: <ollama-name>  [<gguf-file>  [<params>]]  [hf:<owner/repo>]
#   gguf-file  GGUF filename relative to $LLAMA_MODELS_DIR (for --backend llama-server)
#   params     comma-separated llama-server flags
#   hf:        HuggingFace repo for fetch-hf.sh download helper
#
# Hardware: RTX 4090 (~1008 GB/s) + 2× RTX 3090 (~936 GB/s each); all connected by PCIe.
# No NVLink between any pair. For dense tensor-parallel, bandwidth scales as sum across cards
# for generation (memory-bound), but inter-GPU activation transfers add PCIe latency overhead.
# Use: ./gpu-mode.sh multi  before running (sets CUDA_VISIBLE_DEVICES to all 3 GPUs).
#
# What 72 GB unlocks vs 48 GB (2x24gb.txt):
# - Dense 32B: per-GPU KV headroom triples → ctx=131072 feasible (was capped at 32768 on 48 GB)
# - Dense 70B: 3-way split reduces per-GPU weight load; speed up from ~17 to ~22-25 tok/s (est.)
# - qwen2.5:72b-q4: unusable on 48 GB (ctx=16384 max); on 72 GB ctx=65536+ feasible
# - gpt-oss:120b: fully GPU-resident (removes n_cpu_moe CPU offload); ~80-100 tok/s (est.)
# - llama4-scout:17b (109B MoE): GPU-resident for first time; was RAM-bound at 3.3 tok/s on 24 GB
# - qwen3-next:80b: context_256k still OOM (model+256k KV math too tight on 72 GB); keep max_ctx=131072
# - Small/MoE models (<25 GB): no speed or context gain from 3rd GPU; run single-GPU
