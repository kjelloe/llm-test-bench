# ── Carnice-Qwen3.6 MoE 35B A3B (fine-tune by mudler, MTP-trained) ──────────
# Qwen3.6-35B-A3B fine-tune with Multi-Token Prediction (MTP) training.
# GGUF bundles the MTP head for self-speculative decoding (llama.cpp PR #22673, >= commit 255582687).
#
# MTP flag names (for future reference):
#   --spec-type draft-mtp     enable self-speculative decoding (no separate draft model needed)
#   --spec-draft-n-max N      tokens drafted per step (was --draft, now removed)
#   --draft-mtp               NOT a real flag — model card is wrong; use --spec-type draft-mtp
#
# Results (2026-05-19, llama-server 9237 ac76808e4, RTX 3090 24 GB, coding group 15 tasks):
#   WITHOUT spec-type (MTP head loaded but not speculating):
#     carnice:35b  14/15  30.0 tok/s  — fails csv_nordic_property only; passes L5 (dijkstra, hashmap)
#   WITH spec-type=draft-mtp, spec_draft_n_max=4:
#     carnice:35b  13/15  28.3 tok/s  — python_hashmap REGRESSES (model emits _OUTPUT_ONLY_ sentinel
#                                       during speculative rejection/resample); csv_nordic NO_BLOCKS
#     MTP speculation NOT beneficial: acceptance rate low on coding tasks → slower AND lower quality
# Conclusion: run WITHOUT spec flags. 14/15 at 30 tok/s is the correct baseline.
# Speed: ~30 tok/s reflects MTP overhead cost (base qwen3.6:35b-A3B runs at 133 tok/s).
#   MTP head adds ~4× forward-pass overhead; llama.cpp MTP implementation not yet optimized.
# Thinking flag: NOT needed — model achieves L5 without it; adding thinking to MTP fine-tunes
# may cause planning loops. Leave off until confirmed otherwise.

# ── Nemotron-3-Nano 30B-A3B ──────────────────────────────────────────────────
# Hybrid MoE: 23 Mamba-2/MoE layers + 6 Attention layers. 3.5B active / 30B total.
# Q4_K_M ~18 GB — fully GPU-resident on 24 GB with ~6 GB headroom.
# Thinking-capable via chat template flag (same pattern as Qwen3).
# KV cache overhead ≈ 1/5 of a pure transformer (only 6 attention layers accumulate KV):
#   max_ctx=131072 is safe; 256k or higher likely feasible — needs testing.
# Benchmarks (NVIDIA, vs qwen3-coder:30b proxy Qwen3-30B-A3B-Thinking-2507):
#   AIME25 89.1 (vs 85.0), LiveCodeBench 68.3 (vs 66.0), SWE-Bench 38.8 (vs 22.0)
# Results (2026-05-15, llama-server, RTX 3090 24 GB, coding tasks — see non-determinism note):
#   12/15  168.3 tok/s — fails csv_nordic_property, python_expr_eval, python_tokenizer
# Results (2026-05-20, llama-server, RTX 3090 24 GB, full 29-task run):
#   20/29  168.3 tok/s  456s total
#   Coding:   12/15 — fails node_slugify, python_dijkstra, python_hashmap (different 3 than May-15!)
#   L6 step:   3/4  — fails node_para_entities (consistent); passes core/turret/combat
#   l6_full:   FAIL — ES module export failure; model writes class without `export` keyword (2.6k tok)
#   Context:   PASS 8k/16k/32k/64k (171→162→147→116 tok/s); Mamba-2 KV advantage vs transformer
#   Multihop:  FAIL both forward+reverse (2-hop retrieval gap); distractor_notes PASS (1-hop OK)
#   Skips:     context_128k (SKIPPED_CTX, max=65536), context_256k (SKIPPED_VRAM)
# NON-DETERMINISM: both runs score 12/15 coding but fail completely different tasks. Mamba-2
#   recurrent state has floating-point variance even at temperature=0 seed=1. Per-task results
#   on borderline coding tasks are unreliable; only aggregate 12/15 is a stable signal.
# Fastest model tested: 168–181 tok/s; context_64k at 116 tok/s vs qwen3-coder at 76 tok/s.
# Safe limit on 24 GB is ctx=65536. Tested q8_0 and q4_0 KV — both OOM at ctx=131072.
# KV cache precision is not the bottleneck: llama.cpp appears to pre-allocate Mamba-2
# activation/state buffers proportional to context length (not just the 6 attention layers).
# This is a llama.cpp bug — should be filed upstream. max_ctx=65536 is the confirmed safe limit.
